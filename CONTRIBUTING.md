# Contributing

Contributions are welcome! Please read these guidelines before opening a pull request or issue.

## Setup

```bash
git clone https://github.com/henriqueamrl/asaas-php
cd asaas-php
composer install
```

Running `composer install` automatically installs git hooks (pre-commit and commit-msg) via the GrumPHP Composer plugin.

To regenerate the changelog locally, you also need the `git-cliff` binary. Install it via one of:

- macOS/Linux via Homebrew: `brew install git-cliff`
- Cargo: `cargo install git-cliff`
- Download a pre-built binary from <https://github.com/orhun/git-cliff/releases>

See the [git-cliff installation docs](https://git-cliff.org/docs/installation) for full platform-specific instructions.

## Branching Strategy

This project uses GitHub Flow. `main` is the only long-lived branch; all changes arrive through a short-lived feature branch and a pull request.

Branch names follow the pattern `type/short-description`, where `type` is one of:

- `feat` - new feature
- `fix` - bug fix
- `refactor` - code restructuring with no behavior change
- `perf` - performance improvement
- `docs` - documentation only
- `test` - test changes only
- `chore` - maintenance tasks, dependency updates
- `ci` - CI/CD configuration
- `build` - build system changes

Examples:

```
feat/charge-resource
fix/retry-logic
docs/readme-examples
refactor/http-client
```

## Conventional Commits

All commits must follow the conventional commit format:

```
type(scope): description
```

The scope is optional. Examples:

```
feat(customer): add list endpoint with pagination
fix: handle empty response body in HttpClient
refactor(auth): simplify header injection
docs: update README usage examples
```

The same nine types apply as in the branching section: `feat`, `fix`, `refactor`, `perf`, `docs`, `test`, `chore`, `ci`, `build`.

The commit-msg hook (GrumPHP) validates this format locally on every commit.

For breaking changes, include a `BREAKING CHANGE:` footer in the commit body:

```
feat(client): change constructor signature

BREAKING CHANGE: AsaasClient now requires an explicit PSR-18 client argument.
```

The `BREAKING CHANGE:` footer causes a "Breaking Changes" section to appear in the generated changelog.

## Pull Request Workflow

1. Branch off `main` using the `type/short-description` naming convention.
2. Push your branch and open a pull request against `main`.
3. The PR title MUST follow the conventional commit format (for example, `feat(charge): add boleto creation`). This is enforced by the PR Title GitHub Action. PRs are squash-merged, so the PR title becomes the single commit message recorded on `main`.
4. CI must pass before merge: tests across PHP 8.1, 8.2, 8.3, and 8.4; PHPStan level 8; Pint code style; and the PR Title check.

## Hooks (GrumPHP)

Git hooks are managed by GrumPHP and install automatically when you run `composer install` (via the Composer plugin).

- **Pre-commit hook:** runs PHPStan (`composer analyse`) and Pint (`composer format`) on staged PHP files.
- **Commit-msg hook:** validates that the commit message follows the conventional commit format.

Running `git commit --no-verify` bypasses hooks locally, but CI enforces the same checks on every push so they cannot be skipped permanently.

## Standards

- PHP 8.1+
- PSR-12 code style (enforced by Pint)
- PHPStan level 8 (no errors allowed)
- All public methods must have tests
- `declare(strict_types=1)` in every file
- No em-dashes (U+2014) in any tracked file - use `-` instead

## Changelog

`CHANGELOG.md` is auto-generated by [git-cliff](https://git-cliff.org/) from the conventional commit history. **Do not edit `CHANGELOG.md` by hand.** Any manual changes will be overwritten by the release workflow the next time a tag is pushed, and out-of-band entries break the type mapping.

To regenerate the changelog locally:

```bash
git cliff -o CHANGELOG.md
```

Commit type mapping:

| Commit type | Changelog section |
|-------------|-------------------|
| `feat` | Added |
| `fix` | Fixed |
| `refactor`, `perf` | Changed |
| `BREAKING CHANGE:` footer | Breaking Changes |
| `docs`, `test`, `chore`, `ci`, `build` | Excluded |

## Release Process

Releases are triggered by pushing a version tag from `main`:

```bash
git tag v1.2.3
git push --tags
```

The tag push triggers two things in sequence:

1. The existing Tests and Static Analysis workflows run against the tag. The release workflow only fires after both complete successfully.
2. The Release workflow runs git-cliff for the new tag, commits the updated `CHANGELOG.md` back to `main`, and creates a GitHub Release with the generated changelog as the release notes body.

**Note:** Because the release workflow commits the updated `CHANGELOG.md` after the tag is created, the tag itself points to the commit just before the changelog update commit. This is the standard git-cliff release pattern and is expected behavior.
